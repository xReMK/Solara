# High-Level Flow:

- Go CLI: Runs daemon; mnote add #tag -m "text" → goroutine enqueues to channel → background goroutine batches TCP sends to Boot every 5s or on flush.
- Spring Boot: TCP listener service (netty or simple socket) receives batches, persists to Postgres (JPA, tags as JSON array), pushes via WebSocket to UI clients.
- UI: Simple React/Vue (or Thymeleaf) at /ui for search/filter by tag.
- Auth: JWT for CLI/UI; tags auto-categorize (e.g., #philosophy1 appends to existing).
- Future: AWS S3 for note backups (add SDK hands-on)

# wal (write ahead log)
	Receive note via Pipe.
		First: Write it to a local pending.json or wal.log.
		Second: Push it to the Go Memory Queue.
		Third: Send ACK to the Client.
	If the machine dies at Step 2, the file system might recover it. If it dies at Step 1, the Client knows it failed (no ACK)

# Future-Proofing (Scaling)
Because you used a Queue (Channel), scaling is easy:
Parallelism: If you want to send notes to 5 different backends at once, you just start 5 springWorker goroutines
Spring Side: If traffic grows, you can add Redis in front of Postgres to act as a buffer on the Java side

func (p *program) springWorker() {
	for note := range p.queue {
		p.postToSpring(note)
	}
	/*
		By using for note := range p.queue, Go handles the sleeping/waking of the thread for you
		If the queue is empty, the springWorker goroutine is "parked" by the Go scheduler and consumes zero CPU
		When a note arrives, the scheduler wakes it up. This is much more efficient than a while(true) loop in Java
	*/
}

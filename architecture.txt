# "Zero Data Loss" systems never rely purely on application-level batching for durability
To achieve Zero Data Loss without sacrificing performance, you have three professional options:

1. Synchronous WAL (The Gold Standard)
Instead of batching in memory, write every single entry to a Write-Ahead Log (WAL) immediately upon receipt

- The Flow: Go receives text -> Append to WAL file -> Call fsync() (flush to physical disk) -> Return 200 OK to user
- Why it's fast: Sequential disk appends are extremely efficient. Modern SSDs can handle thousands of fsync() calls per second, so 50 req/s is well within limits
  StackOverflow discussions suggest this is the only way to ensure complete consistency against unexpected interruptions
- Recovery: On restart, Go reads the WAL. Any record in the file that isn't marked as "Sent to Spring" is re-queued

2. Embedded Persistent Queue (e.g., SQLite WAL Mode)
Instead of writing a custom file, use an embedded database like SQLite configured in WAL Mode

- Performance: SQLite's WAL mode allows readers and writers to work simultaneously without blocking
- Durability: It manages the atomic "flush to disk" for you. If it crashes mid-write, its internal journaling ensures the database remains uncorrupted
- Scalability: At 50 req/s, SQLite is overkill in terms of capacity, making it a very safe "buffer"

3. External Persistent Broker (e.g., NATS JetStream or RabbitMQ)
If you don't want the Go service to manage files, offload the "Zero Loss" responsibility to a dedicated tool

- The Flow: Go receives text -> Publish to NATS JetStream -> Wait for "Ack" from NATS -> Return 200 OK
- Safety: NATS won't send the "Ack" until it has safely written the message to its own disk. If your Go service crashes, the message is already safe in the broker


Feature		Synchronous WAL (Manual)	SQLite (Embedded DB)	Message Broker (NATS/Rabbit)

Complexity	High (you write the logic)	Medium (simple SQL)	  High (extra infrastructure)
Data Safety	Total				   Total			 Total
Performance	Fastest (no overhead)		  Very Fast               Fast (network hop required)
Maintenance	Manual file cleanup		Easy (SQL deletes)	        Automatic

func (p *program) springWorker() {
	for note := range p.queue {
		p.postToSpring(note)
	}
	/*
		By using for note := range p.queue, Go handles the sleeping/waking of the thread for you
		If the queue is empty, the springWorker goroutine is "parked" by the Go scheduler and consumes zero CPU
		When a note arrives, the scheduler wakes it up. This is much more efficient than a while(true) loop in Java
	*/
}

### The entire purpose of s.Run() is to listen indefinitely for Windows SCM commands
	- If you exited immediately, there'd be no way for SCM to communicate with your service
	- SCM needs the process to stay alive and responding

s.Run() = Blocking (waits for SCM)
p.Start() = Non-blocking (spawns background work)
p.run() = Blocking (endless loop listening on pipe)

s.Run() blocks at event loop
  ├─ Calls p.Start() when needed
  │   └─ p.Start() calls go p.run() (non-blocking)
  │       └─ p.run() blocks in its own for loop
  │
  └─ Returns only when SCM says STOP

The Problem: Distinguishing "Not Provided" from "Zero Value"

In Go, every type has a zero value:
string: ""
bool: false
int: 0

Without pointers:

type UpdateNoteOptions struct {
    Content      string
    Tag          string
    SentToSpring bool
}

// Case 1: User wants to clear the content
opts := UpdateNoteOptions{
    Content: "",
    Tag: "important",
}

// Case 2: User didn't provide content (forgot it)
opts := UpdateNoteOptions{
    Tag: "important",
}

Both look identical inside the function! We can't tell if the user intentionally set Content to empty or just didn't provide it.

With pointers:
type UpdateNoteOptions struct {
    Content      *string
    Tag          *string
    SentToSpring *bool
}

// Case 1: User wants to clear the content
opts := UpdateNoteOptions{
    Content: Ptr(""),
    Tag: Ptr("important"),
}
// Content is a pointer to "", Tag is a pointer to "important"

// Case 2: User didn't provide content
opts := UpdateNoteOptions{
    Tag: Ptr("important"),
}
// Content is nil (not provided), Tag is a pointer to "important"

SQLite in Go: The Embedded Reality
In Go, using modernc.org/sqlite means the database engine lives inside your Go process memory. There is no network overhead or IPC (Inter-Process Communication)

- Data Structure: Like Postgres, SQLite uses B-Trees
- Storage: It stores the entire database in a single cross-platform file (e.g., app.db)
- Persistence: As long as you provide a filename (not :memory:), SQLite is persistent by default
	       Every Commit ensures the data is safely on disk before the function returns

# SQLite WAL Mode: Internals & "Extra" Benefits
In the standard "Rollback" mode, SQLite is a Single-Writer / Single-Reader system. If someone is writing, no one can read.

# How WAL Changes the Internal Mechanics:
When you enable WAL mode in your Go service, SQLite stops overwriting the main .db file directly during a transaction

- The -wal File: All new writes are appended to a sidecar file ending in -wal
- The -shm File (Shared Memory): SQLite creates a "WAL-Index" in RAM (mapped to a file)
	                         This index tells your Go threads: "Hey, if you want Row 5, don't look in the main DB file; the latest version is at Offset X in the WAL file"
- The Checkpoint: Eventually, SQLite moves the data from the WAL file back into the main B-Tree in the .db file

# What this provides you (The "Senior Dev" Perspective):

- Concurrency (The Big Win): Readers do not block writers, and writers do not block readers
	                     In a Go web server, this is essential. Multiple goroutines can read the DB while one goroutine is performing a heavy write
- Performance: Appending to a WAL file is a sequental I/O operation
	       Overwriting B-Tree pages in a main file is random I/O. Sequential is always faster
- Atomicity: It’s actually more robust. If a crash occurs during a checkpoint, the original database file is still intact because the "truth" still exists in the WAL log

# By default, sql.Open creates a pool of connections. While WAL allows multiple readers, it still only allows one writer
  If your pool has 10 connections and 5 of them try to write simultaneously, you'll hit lock contention

# Persistence vs. Performance
In your Go code, I used _pragma=synchronous=NORMAL. Here is what that means for your WAL:
NORMAL: SQLite doesn't wait for the OS to confirm the data is physically on the disk platter after every single WAL write; it relies on the WAL checkpoint for safety. This is the fastest way to use WAL
FULL: SQLite will fsync() every single time you commit. This is slower but "bulletproof
